# 大夜管理和配置页面功能拆解文档

## 一、文档概述

本文档对招行侦测排班系统中"大夜管理和配置"页面进行全面功能拆解，涵盖所有可配置参数、硬编码值、业务规则和功能点。该文档可直接交付给其他开发团队进行代码实施或系统集成。

### 1.1 核心文件清单

- `js/rules/nightShiftConfigRules.js` - 统一配置规则模块（核心配置源）
- `js/rules/nightShiftRules.js` - 已废弃的规则模块（兼容层）
- `js/solvers/nightShift.js` - 夜班排班求解器
- `js/managers/nightShiftManager.js` - 大夜管理器和UI渲染

### 1.2 配置数据流

```
NightShiftConfigRules（内存配置）
    ↓
NightShiftSolver（排班算法）
    ↓
NightShiftManager（UI展示和用户交互）
    ↓
IndexedDB（持久化存储）
```

## 四、功能点详细拆解

### 4.1 配置管理功能

#### 4.1.1 配置加载（NightShiftConfigRules.init）

**功能描述**: 初始化时从 IndexedDB 加载保存的配置，若不存在则使用默认配置

**输入**: 无（自动从 DB 读取）

**输出**: 配置对象 `NightShiftConfigRules.currentConfig`

**依赖**: `DB.loadNightShiftConfig()`

**流程**:

```
开始 → 尝试从DB加载配置
    → 若存在：深度合并默认配置和已保存配置
    → 若不存在：使用默认配置
结束
```

#### 4.1.2 配置保存（NightShiftConfigRules.updateConfig）

**功能描述**: 更新配置并持久化到 IndexedDB

**输入**: `updates` 对象（部分配置更新）

**输出**: 更新后的配置对象

**依赖**: `DB.saveNightShiftConfig()`

**验证**: 调用 `validateConfig()` 确保配置有效

#### 4.1.3 配置验证（NightShiftConfigRules.validateConfig）

**功能描述**: 验证配置的有效性和合理性

**验证项目**:

- 地区配置完整性
- 数值范围合法性（0-5, 3-7等）
- 逻辑一致性（dailyMax >= dailyMin）
- 跨地区配置正确性

**输出**: `{ valid: boolean, errors: string[] }`

### 4.2 排班生成功能

#### 4.2.1 夜班排班主流程（NightShiftSolver.generateNightShiftSchedule）

**功能描述**: 生成完整的大夜排班方案

**输入参数**:

```javascript
{
    staffData: Array,          // 人员数据列表
    scheduleConfig: Object,    // 排班配置 { startDate, endDate, year, month }
    personalRequests: Object,  // 个性化休假需求 { "staffId": { "YYYY-MM-DD": "REQ" } }
    restDays: Object,          // 法定休息日 { "YYYY-MM-DD": true/false }
    rules: Object              // 规则配置（可选，默认使用NightShiftRules）
}
```

**输出结果**:

```javascript
{
    schedule: Object,           // 排班结果 { staffId: { dateStr: 'NIGHT' } }
    mandatoryRestDays: Object,  // 强制休息日 { staffId: [dateStr, ...] }
    stats: Object,              // 统计信息
    validation: Object          // 验证结果
}
```

**核心流程**:

```
1. 获取配置规则
2. 生成日期列表
3. 过滤可用人员（排除哺乳期、孕妇）
4. 按性别分组（上海地区）
5. 人力富足判断
6. 应用女生优先策略
7. 分配夜班（连续/分散）
8. 统计结果
9. 验证排班结果
```

#### 4.2.2 人力富足判断（NightShiftSolver.checkManpowerSufficiency）

**功能描述**: 判断指定地点的人力是否充足

**计算公式**:

```
总需求 = 排班天数 × 每日需求人数
男生供给 = 可用男生数 × 每月标准天数(4天)
女生供给 = 可用女生数 × 每月标准天数(3天)
总供给 = 男生供给 + 女生供给
富足判断 = 总供给 >= 总需求
```

**输入**: 地点人员列表、排班配置、地点配置

**输出**: `{ isSufficient, totalSupply, totalDemand, details }`

#### 4.2.3 休假冲突检测（NightShiftSolver.checkVacationConflict）

**功能描述**: 检测指定日期是否有休假冲突

**冲突类型**:

| 休假类型 | 严格模式 | 跳过LEGAL | 跳过REQ |
|---------|---------|-----------|---------|
| ANNUAL | 冲突 | - | - |
| SICK | 冲突 | - | - |
| LEGAL | 冲突 | 跳过 | - |
| REQ | 冲突 | - | 跳过 |

**优先级**: 严格模式 > 跳过配置

#### 4.2.4 生理期处理（NightShiftSolver.getMenstrualPeriod）

**功能描述**: 获取员工生理期需要避开的日期

**处理逻辑**:

- 根据 `menstrualPeriodType`（'upper'/'上' 或 'lower'/'下'）确定时间段
- 上半月：日期 1-15
- 下半月：日期 16-月末

### 4.3 人员过滤功能

#### 4.3.1 基础条件过滤（NightShiftSolver.filterAvailableStaff）

**排除条件**:

- `isLactating === true` 或 `lactating === true`（哺乳期）
- `isPregnant === true` 或 `pregnant === true`（孕妇）

**控制开关**: `constraints.checkBasicEligibility`

#### 4.3.2 可用日期过滤（NightShiftSolver.findAvailableContinuousPeriod）

**排除条件**:

- 休息日（`restDays[dateStr] === true`）
- 固定节假日（调用 `HolidayManager.isFixedHoliday()`）
- 休假冲突（调用 `checkVacationConflict()`）
- 生理期日期
- 已分配日期（`existingSchedule[dateStr]`）
- 强制休息日（`mandatoryRestList.includes(dateStr)`）
- 每天人数已达上限

### 4.4 统计与验证功能

#### 4.4.1 排班统计（NightShiftSolver.calculateStats）

**统计项目**:

- `totalNightShifts`: 总夜班数
- `staffNightShiftCounts`: 每人夜班天数 { staffId: count }
- `locationCounts`: 按地点统计 { location: count }

#### 4.4.2 排班验证（NightShiftSolver.validateNightShiftSchedule）

**验证项目**:

1. **硬上限检查**: 每人分配天数 <= maxDaysPerMonth
2. **每天人数检查**: 每天人数 <= 跨地区最大总人数(4)
3. **连续天数检查**: 连续夜班天数 <= 最大连续天数

**输出**: `{ errors: [], warnings: [], isValid: boolean }`

## 五、配置影响矩阵

### 5.1 配置项影响范围汇总

| 配置类别 | 影响人员 | 影响天数 | 影响约束 | 影响公平性 |
|---------|---------|---------|---------|-----------|
| 地区配置（dailyMin/Max） | ✅ | ✅ | ✅ | - |
| 连续天数（consecutiveDays） | - | ✅ | ✅ | - |
| 月度上限（maxDaysPerMonth） | ✅ | ✅ | - | ✅ |
| 人力计算（richThreshold） | - | ✅ | - | ✅ |
| 休假冲突（vacationSkip） | ✅ | ✅ | ✅ | - |
| 生理期（checkMenstrualPeriod） | ✅（女） | ✅ | ✅ | - |
| 排班模式（arrangementMode） | - | ✅ | ✅ | ✅ |
| 女生优先（femalePriority） | ✅（女） | ✅ | - | ✅ |
| 上月权重（lastMonthWeight） | ✅ | ✅ | - | ✅ |

### 5.2 关键配置变更影响

#### 5.2.1 dailyMax 从 2 改为 3

**直接影响**:

- 每天可排大夜人数增加 1 人
- 总夜班数增加约 30%（假设30天排班周期）
- 需要检查人力是否充足

**间接影响**:

- 可能需要调整 monthlyMaxDaysPerMonth
- 可能需要调整连续天数限制

#### 5.2.2 maleMaxDaysPerMonth 从 4 改为 5

**直接影响**:

- 男生每月最多可排 5 天大夜
- 人力供给增加 25%
- 可能影响女生排班机会

**间接影响**:

- 需要重新评估人力富足判断
- 可能触发公平性调整

#### 5.2.3 arrangementMode 从 'continuous' 改为 'distributed'

**直接影响**:

- 排班从连续模式变为分散模式
- 两次大夜间隔至少 minIntervalDays（默认7天）
- 月度排班分布更均匀

**间接影响**:

- 可能减少连续休息日
- 可能增加排班难度（日期选择受限）

## 九、实施建议

### 9.1 配置界面设计建议

1. **分组展示**: 将配置按功能分组（地区配置、约束规则、优先级策略）
2. **实时预览**: 修改配置后显示预估影响（如预计总夜班数、人力供需比）
3. **配置对比**: 支持保存多套配置并对比效果
4. **导入导出**: 支持配置 JSON 导入导出

### 9.2 性能优化建议

1. **配置缓存**: 配置读取频繁，建议增加内存缓存
2. **增量计算**: 人力判断可增量更新，无需每次全量计算
3. **Web Worker**: 排班生成是 CPU 密集型，建议移至 Web Worker

### 9.3 测试要点

1. **边界测试**: 测试各种极限配置组合（0人/天、7天连续等）
2. **冲突测试**: 测试各种约束冲突场景
3. **性能测试**: 测试大批量人员（100+）的排班生成性能
4. **持久化测试**: 测试配置保存/加载的正确性
